# 《深岩银河幸存者》游戏助手 — 简单 PRD

> 文档类型：简单 PRD（确认用，非详细规格）
> 作者：产品经理 许清楚（Xu）
> 状态：待用户确认
> 数据底稿：`references/data-collected.md`（来自 Steam 社区攻略，已抓取汇总）

---

## 一、产品目标

**一句话定位**：一个纯前端的《深岩银河幸存者》(DRG: Survivor) 速查 Web 应用，帮玩家在游戏中随时检索成就、武器与装备的关键信息，无需联网、即开即用。

**用户价值**：
- 把分散在多个 Steam 社区攻略里的信息（成就解锁条件、武器超频模块与评级、装备效果）聚合到一个可搜索、可筛选的界面；
- 打游戏时快速确认"这把武器怎么超频、评级如何""这个成就怎么解锁""这件装备该不该拿"，减少切屏查攻略的时间；
- 支持中/英文名与标签检索，照顾不同习惯的玩家；评级与标签可视化，一眼看清强弱。

---

## 二、用户故事

1. **作为新手玩家**，我想按职业查看所有武器及其评级（S/A/B/C），以便快速决定主玩哪把枪、优先升哪把。
2. **作为冲成就的玩家**，我想按 12 个分类筛选成就并看到解锁条件，以便规划"还差哪个成就没拿"。
3. **作为配装玩家**，我想搜索某个标签（如"冰冻""无人机""炮塔"）找到相关武器与装备，以便围绕流派组队。
4. **作为速通/挑战玩家**，我想用关键词（中/英文）模糊搜索（如"爆头""Nitra""护甲"）秒定位对应条目，以便不打断节奏。
5. **作为硬核玩家**，我想在武器卡片上直接看到黄/红超频模块与强度评级，以便对比不同武器在同职业下的优劣。

## 三、需求池

### P0（必须有，上线即交付）
- **三大模块完整展示**
  - 成就：约 110 条，字段 = 英文名 + 中文译名 + 解锁条件 + 分类（12 类）；生物群系类需呈现 H5 / 精通 / 真精通 三种。
  - 武器：约 45–50 把，字段 = 英文名 + 中文名 + 职业(class) + 标签(tags) + 黄色超频(6/12级) + 红色超频(18级) + 强度评级(S/A/B/C/-)。
  - 装备：约 21 件，字段 = 名称 + 类型 + 效果 + 来源(source，区分"局内附加"与"成就解锁"两类)。
- **全局搜索**：顶部搜索框，按中英文名 / 标签 / 条件模糊匹配，跨三大模块即时过滤。
- **分类筛选**：
  - 成就按 12 个分类筛选；
  - 武器按职业（Scout/Gunner/Engineer/Driller）+ 强度评级（S/A/B/C/-）筛选；
  - 装备按类型（发育/拾取/生存/武器/战力/直伤/闪避/暴击…）+ 来源（局内 / 成就解锁）筛选。
- **即时生效**：搜索与筛选输入后实时更新结果，无需点"查询"按钮。
- **数据完整性**：所有底稿数据需 100% 录入，不遗漏、不杜撰（评级仅来自攻略作者的主观评价，需如实标注）。

### P1（很重要，确认后优先做）
- **评级可视化**：S 金色徽章、A/B/C 对应色阶徽章、"-"显示"未评级"，避免纯文字难辨。
- **标签 chips**：武器与装备的标签以彩色 chip 呈现，可点击作为快捷筛选条件。
- **响应式布局**：桌面端多列卡片网格，移动端单列堆叠，保证手机上也能查。
- **数据来源标注**：页面/模块内标注数据来自 Steam 社区攻略，并注明抓取日期与"评级为攻略作者主观评价"的免责说明。

### P2（可选，资源允许再做）
- **收藏 / 对比**：收藏常用武器或成就，武器间并排对比超频与评级。
- **导出**：将筛选结果导出为图片/CSV 分享。
- **多语言**：UI 文案中英双语切换（数据本身已含中英双字段）。
- **深色主题切换**：明/暗主题切换并记忆偏好。
- **疑难成就高亮**：将"全球达成率<50%"的疑难成就单独标注（底稿已列出十余条）。

### 4.2 各模块卡片字段与控件位置

| 模块 | 卡片字段 | 筛选控件（位于筛选区） | 备注 |
|---|---|---|---|
| 成就 | 中文译名、英文名、解锁条件、分类标签 | 分类下拉/12 类 chips | 生物群系类卡片需区分 H5 / 精通 / 真精通 |
| 武器 | 中文名、英文名、职业、标签 chips、黄色超频、红色超频、评级徽章 | 职业下拉、评级下拉、标签 chips | 同武器跨职业（如 Plasma Carbine）按各自条目展示 |
| 装备 | 名称、类型 chip、效果、来源标签 | 类型下拉、来源下拉（局内/成就解锁） | 成就解锁类可附"解锁成就"字段 |

### 4.3 交互要点
- 顶部搜索框 + Tab 常驻；搜索与筛选组合作用（AND 关系）。
- 标签 chip 可点击 → 自动填入搜索/筛选（P1，提升检索效率）。
- 卡片采用 MUI `Card` + Tailwind 布局，网格用 `Grid`/`flex-wrap` 实现响应式。

## 五、待确认问题（需用户拍板）

1. **UI 文案语言**：以**中文为主**，还是**中英双语同显**（界面按钮、分类名也中英对照）？数据本身已含中英双字段，主要是界面外壳文案的取舍。
2. **明暗主题**：默认**浅色**即可，还是要**深色**（游戏玩家多偏好暗色）？是否做 P2 的主题切换？
3. **数据来源/抓取日期标注**：是否需要页脚标注"数据来源 + 抓取日期 + 评级主观免责"？建议**保留**（关系到可信度）。
4. **P2 功能取舍**：收藏/对比、导出、多语言、深色切换中，哪些**本次就要**？建议默认只做 P0+P1，P2 待定。
5. **部署方式**：是部署为**可公网访问的静态站点**（如 CloudStudio/GitHub Pages），还是本地 `npm run dev` 自用即可？影响是否需加部署配置。
6. **疑难成就高亮**：是否要把"全球达成率<50%"的疑难成就在成就模块单独高亮/加筛选？（底稿已备数据，成本低）

## 六、关于"走快速模式"的判断依据

1. **规模小**：单页应用，三大 Tab + 搜索/筛选，预期 ≤10 个源文件（3 个数据模块 + 少量组件 + 入口），属轻量前端。
2. **数据静态且已就绪**：底稿已结构化（中/英双字段、分类、评级），无需爬取、无需后端/数据库，纯前端内嵌即可。
3. **技术栈成熟**：Vite + React + MUI + Tailwind 为标准组合，无新技术风险。
4. **需求聚焦**：核心价值=查询（搜索+筛选+展示），无复杂业务流、无登录/支付/实时通信。
5. **可确认后快速实现**：需求边界清晰，确认 PRD 后即可直接进入开发，无需多轮返工。

→ 结论：**建议走快速模式**，确认上述待确认问题后由工程师直接实现，预计工作量小、周期短。
